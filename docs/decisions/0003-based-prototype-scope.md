# 0003 — Based prototype scope and invariant remap

## Status
Accepted (prototype phase). Supersedes the generic SaaS invariants in
`architecture.md` **for the prototype only**; they remain the production target.

## Context
The repo was scaffolded as a generic multi-tenant SaaS (`AGENTS.md`,
`architecture.md`): Fastify API with auth, RBAC, billing/Stripe, Postgres, and
"non-negotiable" tenant-isolation invariants. The actual product, defined in
`BASED_PROTOTYPE_BRIEF.md`, is different: a **frontend-forward, demoable
prototype** of an AI "host" for live-stream discovery, with the expensive
perception backend **mocked behind typed contracts** (§6:
`PerceptionEvent → RankedFeed → HostDirective`). The brief (§15) explicitly puts
**auth, accounts, and persistence out of scope**. The two cannot both drive the
build; the navigator chose to build Based per the brief.

## Decision
- **Build the Based prototype** following the brief's milestones M0–M4 (M5 stretch).
- **Remap the invariants** the TDD loop must prove with tests, from the generic
  SaaS set to the Based set:
  1. **Spoiler-safety** (the tenant-isolation analog): every `HostDirective` is
     `spoilerSafe: true`; the host never names an outcome before the cut lands.
  2. **Silence budget:** idle by default; speaking is rate-limited.
  3. **Contracts-as-seam:** the UI consumes only `HostDirective` + channel state;
     swapping mock perception for real never touches the UI.
  4. **Cost-gating:** narration/LLM fires only on events, never continuously.
  5. **Official embeds only / route value to the source.**
  6. **Secrets from env:** `GEMINI_API_KEY` lives only in `backend/.env`
     (gitignored); never committed or logged.
- **Narration** is generated by **Gemini 3.1 Flash Live** behind a thin Fastify
  proxy (`POST /narrate`) so the key stays server-side and we keep a real,
  testable backend contract. (Confirm the exact model id/endpoint against Google's
  docs at implementation time.)
- The **host loop runs client-side** in the prototype (pure logic, easily tested),
  matching brief §8 ("the mock backend pushes `HostDirective`s onto the bus"). The
  only real server component is the narration proxy.
- **§6 contracts live in `frontend/src/contracts/`** (types only); the backend
  validates its own request with a local zod schema (no cross-workspace runtime
  coupling). Promote to a shared package only if a second backend consumer appears.

## Consequences
- The generic SaaS invariants (multi-tenancy, AuthN/AuthZ, billing, migrations) are
  **deferred**, not deleted; `architecture.md` remains the eventual production
  shape for when persistence/perception land.
- The TDD discipline is unchanged — we still prove each invariant with a test
  before shipping the path that needs it; only the list of invariants changed.
- Cross-layer order is brief-driven: most work is frontend-first; the backend
  appears at M3; one e2e journey is added once the frontend journey exists.

## Alternatives considered
- **Follow the generic SaaS protocol literally** (auth/tenancy/billing/Postgres
  first): contradicts the brief's "see it live this week" goal and §15; rejected.
- **Enforce the SaaS invariants on top of Based**: adds scope (tenants/auth) the
  brief deferred, for no prototype value; rejected.
